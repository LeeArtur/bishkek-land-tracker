Варианты архитектуры

Выбери подход, который лучше всего подходит под твои цели

A

Streamlit / Plotly Dash — All-in-one Python

Стек: Python · Playwright/BeautifulSoup · SQLite · Streamlit или Dash

Как работает: Весь стек — один Python-проект. Парсер собирает данные, кладёт в SQLite, Streamlit строит дашборд прямо из базы. Никакого отдельного фронтенда.

Плюсы

  • Самый быстрый старт (2-3 дня)
  • Один язык, один проект
  • Встроенные графики (Plotly)
  • Легко деплоить локально

Минусы

  • Ограниченная кастомизация UI
  • Менее гибкий дашборд
  • Медленнее при сложных фильтрах

Лучше всего если хочешь быстро запустить и не заморачиваться с фронтом

B

FastAPI + React — Классический стек ⭐ Рекомендую

Стек: Python · Playwright · SQLite/PostgreSQL · FastAPI · React + Recharts

Как работает: Python-парсер + FastAPI отдают REST API. React-фронтенд строит красивый дашборд с картой районов, графиками трендов, фильтрами. Cron запускает парсинг каждую ночь.

Плюсы

  • Полная свобода дизайна UI
  • Быстрый и отзывчивый интерфейс
  • Легко добавлять фичи позже
  • Стандартный Production-ready подход

Минусы

  • Нужно знать React (или учиться)
  • 4-7 дней до рабочего MVP
  • Два отдельных сервиса

Лучший баланс гибкости и скорости для личного инструмента

C

Next.js Full-Stack — Один JS-проект

Стек: Node.js · Puppeteer/Playwright · SQLite (Prisma) · Next.js · Tailwind · Recharts

Как работает: Next.js API Routes обрабатывают парсинг и данные. Фронт и бэк в одном репозитории. Cron через node-cron или Vercel Cron.

Плюсы

  • Один язык (TypeScript везде)
  • Современный стек, легко деплоить на Vercel
  • Server Components — быстрый рендеринг

Минусы

  • Парсинг на JS сложнее чем на Python
  • Puppeteer тяжелее Playwright для Python
  • Нет экосистемы pandas/numpy для аналитики

Хорош если уже знаешь JS/React и хочешь один монорепо
